## Working Rules

- Preserve modules first, remove only what violates the PHI boundary
- Reuse the session-boundary pattern already established in `DocumentationStudio` and `HandoutMaker`
- Do not reintroduce patient-linked persistence
- Do not use Graph unless SharePoint-native paths fail the real requirement
- Keep docs, implementation, and validation in sync in the same pass
- Do not add new Gemini-specific implementation work on the production path
- Centralize AI provider changes in the service layer before changing feature modules one by one

## Architecture Rules From Microsoft Learn

- Use SPFx as the real production shell for SharePoint. Do not treat the portal as a standalone SPA divorced from SharePoint context.
- Prefer a Single Part App Page style experience for this portal so navigation, app shell, and permissions stay coherent.
- Prefer SharePoint-native APIs for SharePoint data. Use the SharePoint context and `SPHttpClient` first when the data lives in SharePoint.
- Use `AadHttpClient` for Entra ID-secured APIs other than SharePoint-native endpoints.
- Use `MSGraphClientV3` only when the requirement is truly Microsoft Graph-specific.
- Do not use MSAL directly inside SPFx for these calls when SPFx-native clients already cover the scenario. Microsoft Learn recommends `AadHttpClient` / `MSGraphClientV3`.
- Treat every Entra permission request as a tenant-level decision. Keep scopes minimal and document why each one exists.
- Keep PHI out of property-pane configuration, package manifests, default props, and durable browser storage.
- Prefer session memory for sensitive input and sanitized durable artifacts for anything that must persist.

## API Decision Rules

When implementing data access, use this order:

1. SharePoint data in lists/libraries/site pages:
   use SharePoint context plus `SPHttpClient`
2. Entra-secured custom API or first-party REST API:
   use `AadHttpClient`
3. Graph-only capability:
   use `MSGraphClientV3`

Graph is not the default answer. It is the fallback when SharePoint-native access does not meet the real requirement.

## AI Replacement Rules

- Gemini/Google Cloud is no longer the intended production provider for this portal.
- New AI work must target Microsoft enterprise Copilot-aligned architecture.
- For custom conversational experiences inside this portal, prefer Microsoft 365 Copilot APIs or a custom engine agent path over direct third-party browser-side model calls.
- For agentic workflows, connectors, and enterprise knowledge orchestration, prefer Copilot Studio patterns where they fit the requirement.
- Do not assume Microsoft 365 Copilot can replace every Gemini capability one-for-one.
- Microsoft Learn currently documents the Chat API as text-only and notes that it doesn't support graphic art or long-running content-generation skills. Treat image/video/TTS replacements as separate design decisions.
- Keep a provider seam in `ai-service` so UI modules call stable portal-facing methods rather than provider-specific SDKs.
- Treat legacy cloud AI as migration-only for media and leftover edge cases, not as the default text runtime.

## Enterprise Copilot FAQ

### What should replace Gemini in this portal?

The default production direction is:
- SharePoint/SPFx shell for UI
- Microsoft enterprise Copilot-aligned service boundary for text and grounded reasoning
- local/browser fallbacks or separate Microsoft-approved paths for features the Copilot Chat API does not cover directly

### Can Microsoft 365 Copilot Chat API replace all current Gemini features?

No. Microsoft Learn documents the Chat API as preview, text-only, and not supporting graphic art or long-running tasks. Do not design as if it is a full replacement for image, video, or TTS features.

### What is the best fit for an external SharePoint portal?

For an app or portal experience outside the native Microsoft 365 Copilot surfaces, Microsoft Learn points to Copilot APIs for custom applications and custom engine agents for external apps/websites.

### When should Copilot Studio be used?

Use Copilot Studio when the requirement is better served by enterprise knowledge, actions, connectors, and low-code/managed orchestration. Use a custom engine agent or app API path when the portal needs deeper custom UX or orchestration control.

## Deployment And Packaging Guidance

- Default to packaging the portal as a tenant app-catalog SPFx solution.
- Use tenant-scoped deployment only if the solution does not rely on Feature Framework provisioning for required site assets.
- Microsoft Learn notes that `skipFeatureDeployment` is not the right choice when solution-specific SharePoint assets must be provisioned.
- Default to including client-side assets in the package unless there is a proven need for CDN-hosted assets.
- If external CDN hosting is introduced later, document:
  - why SharePoint-hosted assets are insufficient
  - the chosen CDN base path
  - cache behavior
  - release steps for production packaging

## Security And Permissions Guidance

- Minimize requested permissions at all times.
- Record every new Entra permission request in docs before asking an admin to approve it.
- Prefer site-scoped SharePoint access patterns over tenant-wide data access patterns whenever architecture allows.
- Treat browser persistence as durable storage. If the content would be unsafe in localStorage or IndexedDB, do not store it there.
- Sanitization is a fallback, not permission to capture more PHI than necessary.

## SharePoint-Specific Tips

- Use the property pane only for stable configuration that belongs to the site or the web part instance, not for live clinical session state.
- If the app is later decomposed into multiple SPFx parts on a page, prefer standard SPFx dynamic-data patterns for cross-part wiring instead of custom ad hoc event plumbing.
- Keep app-shell routing and patient/session state separate. Route state can persist; sensitive session content should not.
- Validate production bundles after major changes because SharePoint-hosted SPFx experiences are sensitive to oversized client bundles and slow first load.

## FAQ

### Should we use Graph for SharePoint files or lists?

Not by default. If the data is fundamentally SharePoint content, start with SharePoint-native APIs and SPFx context. Move to Graph only if the requirement is specifically Graph-only or cross-workload.

### Should we use domain-isolated web parts for extra security?

No. Microsoft Learn says SharePoint Framework domain-isolated web parts were fully retired on April 2, 2026. Do not design around domain isolation for this portal.

### Should we use tenant-scoped deployment?

Only if the solution is meant to be broadly available and does not depend on Feature Framework provisioning for required assets like lists or other site artifacts.

### Should we host assets in a CDN?

Only if there is a concrete performance or operational reason. Learn documents CDN deployment as an option, not a requirement. SharePoint-hosted assets are acceptable unless there is a proven need to externalize them.

### Where should session-sensitive clinical work live?

In memory during the session. If something must persist, persist a sanitized artifact or non-PHI derivative only.
